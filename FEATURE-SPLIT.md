# 기능 분리: Frontend vs Backend

2차 과제는 `localStorage`에 데이터를 저장했다. 이번에는 FastAPI 서버가
데이터를 관리한다. 그래서 기존에 프론트가 떠안던 일 중 **데이터의 저장·조회·수정·삭제**는
백엔드로 넘어가고, 프론트는 **화면 표현과 사용자 입력 처리**에 집중한다.

## 핵심 전환

| 2차 과제 (localStorage) | 3차 과제 (서버 API) |
| --- | --- |
| 브라우저 `localStorage`에 직접 read/write | FastAPI + SQLite가 데이터의 단일 출처 |
| 새로고침 시 브라우저 저장소에서 복구 | 페이지 진입 시 서버에서 fetch |
| 데이터 형태/검증을 프론트가 책임 | Pydantic 스키마로 백엔드가 검증 |

## 백엔드(FastAPI)로 넘어가는 일

- Todo 데이터 영속화 (SQLite, SQLAlchemy)
- CRUD API 제공
  - `GET /todos` — 목록 조회
  - `POST /todos` — 생성
  - `PUT/PATCH /todos/{id}` — 수정(내용·완료 토글)
  - `DELETE /todos/{id}` — 삭제
- 요청/응답 데이터 검증 (Pydantic 스키마)
- 데이터 형태(id 발급, 생성시각 등)의 최종 책임

## 프론트엔드(Next.js)에 남는 일

- 주간 캘린더 렌더링 / 이전·다음 주 이동
- 날짜 선택, 날짜별 개수 표시
- 검색어·상태 필터 등 **화면 표현용 가공** (서버 데이터를 받아 UI에서 거르기)
- 입력 폼(추가/수정) UI 와 사용자 인터랙션
- 서버 API 호출 (`route.ts` 프록시 + Server Actions / Axios)

## 경계가 애매한 부분 (판단)

- **검색·상태·날짜 필터**: 서버 쿼리 파라미터로 넘길 수도 있고, 프론트에서
  걸러낼 수도 있다. 이번 과제 규모(SQLite, 단일 사용자)에서는 목록 전체를
  받아 **프론트에서 필터링**하는 쪽이 단순하다고 판단. 데이터가 커지면
  서버 필터링으로 옮긴다.
- **정렬/개수 집계**: 표현 목적이므로 프론트에서 처리. 단, 정합성이 중요해지면
  서버로 이관 가능.

## 데이터 흐름 요약

```
[브라우저 UI]  ──입력──▶  [Next.js route.ts / Server Action]  ──HTTP(Axios)──▶  [FastAPI]  ──▶  [SQLite]
      ▲                                                                                          │
      └──────────────────────────  화면 렌더링 (서버에서 받은 데이터)  ◀──────────────────────────┘
```
