# study-post — 학습 정리 → 사람 톤 글 → GitHub Issue

멘티의 거친 학습 메모를 받아, `writing-guide.md` 규칙대로 사람이 쓴 톤의
학습 정리 글로 변환한다. 최종 출력은 GitHub Issue에 그대로 복붙할 형식.
게시는 사용자가 직접 한다 (자동 생성하지 않음).

## 입력
- 사용자가 `note-template.md`를 채워서 붙여넣는다.
- 입력이 안 보이면, `note-template.md` 내용을 보여주고 채워달라고 요청한다.

## 게이트 (글 쓰기 전 반드시 확인)
다음 두 항목이 비어 있으면 **글을 쓰지 말고** 먼저 1~2개 되묻는다:
- "막혔던 점 / 시행착오"
- "실제 코드 / 파일명 / 숫자 등 근거"

이 둘이 없으면 결국 generic AI 글이 된다. (writing-guide.md 4번)

## 워크플로
1. 같은 폴더의 `writing-guide.md`를 읽어 규칙을 로드한다.
2. [Outline] 입력을 3단 구성 개요로 잡는다 (도입 → 핵심 → 정리/판단).
3. [Drafting] 개요별로 입력의 실제 코드·경험을 녹여 초안 작성.
4. [Refinement] writing-guide.md 2·3번으로 자가 점검:
   - 한국어 AI 티 단어 치환
   - 문장 길이 들쭉날쭉하게 (짧은 문장 최소 1회)
   - '것/점/의' 3회 이상 반복 제거
5. [Self-check] "나만 쓸 수 있는 문장"이 0개면 4번으로 되돌아간다.

## 출력 형식 (이 형태로만)
GitHub Issue는 Title 칸과 Body 칸이 분리돼 있다. 사용자가 렌더링된 글이
아니라 "원본 마크다운"을 복사해 붙일 수 있도록, 본문은 코드블록 안에 담는다.

- 제목과 본문을 분리해 제시한다. "## 제목 제안 / ## 본문" 같은 라벨은
  복붙 대상(코드블록) 안에 넣지 않는다.
- 본문은 코드펜스(```jsx 등)를 포함하므로, 바깥 래퍼는 backtick 4개(````)로
  열고 닫아 안쪽 ``` 가 깨지지 않게 한다. 래퍼 언어는 ````markdown.
- 본문 안에 "참고 자료" 섹션(공식/1차 문서 링크)을 포함한다.

출력 구조:
1. "Title 칸에 넣을 것:" + backtick 블록에 제목 한 줄
2. "Body 칸에 넣을 것 (블록 안쪽을 그대로 복사):" + ````markdown 블록에
   본문 전체(### 소제목들 + 참고 자료 섹션)
3. 마지막에 한 줄: "Preview 탭으로 확인 후 게시하세요."
4. (선택) 라벨 제안은 블록 바깥에 따로 적는다.

## 참고 자료 규칙
- 반드시 1차 출처(공식 문서)를 우선한다: React → react.dev, 웹 API → MDN.
- URL을 지어내지 말 것. 불확실하면 WebFetch로 페이지·anchor를 확인하고 단다.
- 블로그/요약 글보다 공식 문서를 우선. 개념마다 1개씩이면 충분.

출력 후 한 줄 안내: "검토하고 GitHub Issue 탭에 붙여넣으세요."
사용자가 명시적으로 요청하기 전에는 `gh` 등으로 이슈를 자동 생성하지 않는다.
