# Writing Guide (Human-Tone, 멘토-멘티 과제 정리용)

## 0. 목적
멘티가 학습한 내용을 정리해 멘토에게 제출하는 글. 멘토가 이 글로
멘티의 이해를 '검증'한다. 따라서 개념을 정답처럼 단정하는 글이 아니라,
"내가 이 개념을 이렇게 이해했다"를 서술하는 글이어야 한다.
"잘 다듬어진 AI 글"이 아니라 "직접 이해하고 겪은 사람이 쓴 글"로 읽히는 것이 목표.
→ 화려함보다 '구체성'과 '본인 흔적'이 우선.

## 1. Persona
- Tone: 개발 기술 정리 글. 담백하고 정확하게. 회고/일기체 아님.
- 종결: 존댓말(~합니다 / ~입니다). 반말 금지.
- Reader: 같은 주제를 공부하는 동료 개발자. 설명하되 감상은 빼고 정보 위주.
- Voice: 직접 구현해본 사람의 1인칭. 단, "막혔다 / 헤맸다" 같은 감정 서술 대신
  "처음엔 A로 구현했으나 B 문제로 변경" 식의 사실 기술.
- 관점: 개념을 정의·단정하지 말고 "내가 이해한 바"로 서술한다.
  "제가 생각하는 / 제가 생각한"을 핵심 프레이밍으로 쓴다.
  - 단정형("~는 ~이다 / ~가 핵심이다") → "제가 생각하는 ~는 ~입니다 /
    제가 이렇게 이해했습니다"
  - 객관 서술("흐름이 이어집니다 / ~가 그 근거입니다") 금지 →
    "제가 생각한 흐름은 이렇습니다 / 제가 생각하는 근거는 ~입니다"
  - 각 항목은 "제가 이 개념을 이렇게 이해했다 + 제가 생각하는 근거(코드/경험)" 순서로.
  - 멘토가 검증할 수 있도록, 확신이 약한 부분은 솔직히 표시한다.

## 2. 한국어 AI 티 금지 목록 (Ban Rules)
다음이 보이면 사람 글이 아님. 자동으로 걸러낼 것.
- 진부한 연결어: "아시다시피", "결론적으로", "첫째로", "또한", "나아가",
  "뿐만 아니라"로 문단을 시작하지 말 것
- 텅 빈 형용사: "다양한", "효율적인", "중요한", "강력한", "손쉽게",
  "효과적으로" — 구체적 수치/예시로 대체 (예: "다양한 기능" → "필터 3종")
- 공허한 마무리 문장: "중요한 역할을 합니다", "필수적인 요소입니다",
  "~라고 할 수 있습니다" 금지
- 명사화 남발: 한 문장에 '~것', '~점', '~의' 3회 이상 금지
- 균일한 리듬: 모든 문장이 비슷한 길이/구조면 안 됨 (3번 항목 참조)
- 과한 대칭: 불릿마다 토씨까지 똑같은 구조로 끝맺지 말 것
- 반말 금지: "~다 / ~더라 / ~거든 / ~셈이다" → "~합니다 / ~입니다"로
- 회고·일기체 금지: "헤맸다", "~더라", "끝.", 감정 서술, 다짐("다음엔 ~해야겠다")
- 과한 문학적 비유 금지: "한 바퀴가 맞물린다" 류의 수사 빼고 사실만
- 강조용 볼드(`**...**`) 금지: 강조는 마크업이 아니라 문장으로 한다

## 3. 문장 리듬 (Burstiness)
- 문장 길이를 일부러 들쭉날쭉하게. 짧은 문장과 긴 문장을 섞을 것.
- 단조로운 "~합니다" 반복을 피하고, 명사형 마무리·짧은 단정문을 섞을 것.
- 단, 구어체 종결("~거든요", "~더라고요")은 쓰지 않는다. 존댓말 기술문 유지.

## 4. 본인 흔적 주입 (가장 중요)
글에 아래 중 최소 2개를 반드시 포함:
- 처음에 잘못 이해했다가 고친 지점 ("처음엔 ~인 줄 알았는데 알고 보니")
- 실제 코드/숫자/파일명 등 구체적 근거
- 직접 비교해보고 든 판단 ("A 대신 B를 택한 이유는")
- 아직 헷갈리거나 확신 없는 부분 (솔직하게 남겨도 됨)

## 5. 자가 점검
- 마지막 질문: "이 글에서 나만 쓸 수 있는 문장이 몇 개인가?"
  0개면 다시 쓸 것.
